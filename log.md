# 설계 변경 기록

`campus-notice-feed`의 주요 설계 결정과 그 근거를 시간순으로 남긴다.
(현재는 KHU EE 전용, 추후 타 학과/학교로 일반화 예정)

## 2026-07-11

### 1. 이슈 생성 경로를 파일 전달 방식으로 (보안)

**바뀐 것:** 게시글 제목/본문을 `GITHUB_OUTPUT`으로 넘겨 워크플로에서
`${{ steps.check.outputs.* }}`로 인라인하던 것을 →
파이썬이 `issue_title.txt` / `issue_body.md` 파일로 쓰고,
github-script가 `fs.readFileSync`로 읽도록 변경. `GITHUB_OUTPUT`에는
`has_new` 불리언만 남김.

**근거:** 게시글 텍스트는 외부 통제 불가 입력이다. `${{ }}`로 인라인하면
스크립트 인젝션이 가능하고(제목에 백틱/`${...}` 삽입 시 임의 JS 실행),
본문을 heredoc(`body<<EOF`)으로 넘기면 본문에 `EOF` 줄이 있을 때 출력이
깨진다. 파일 전달은 두 취약점을 한 번에 제거한다. env 방식은 인젝션만
막고 heredoc 문제는 남아서 최선이 아니었다.

### 2. 관심 키워드를 외부 파일로 분리

**바뀐 것:** 코드 안 `KEYWORDS` 리스트 → `keywords.txt` 파일.
한 줄에 하나, `#` 주석·빈 줄 무시. 파일이 없거나 비면 코드의
`DEFAULT_KEYWORDS`로 폴백.

**근거:** 키워드는 자주 바뀌는 사용자 데이터인데 코드에 박혀 있으면
수정할 때마다 소스를 건드려야 한다. 평문 파일이면 GitHub 웹에서 바로
편집·커밋 가능하고 의존성도 안 늘어난다. 지금은 '평평한 목록'이라
평문이 최적. 키워드별 가중치·카테고리·채널 라우팅이 필요해지면 그때
TOML(파이썬 3.11+ 내장 `tomllib`, 무의존성)로 승급.

### 3. 마크업 변경 감지 (파서 건강검사)

**바뀐 것:** `fetch_board`가 `(rows, candidates)`를 반환하도록 변경.
`candidates` = td 2개 이상인 '데이터 행' 수(헤더·이미지 배너·"게시물
없음" 행 제외). `main`에서 두 신호로 조용한 실패를 감지:
- `candidates == 0` → 구조가 통째로 바뀌어 데이터 행 자체가 사라짐
  (예: table → div). 빈 게시판 가능성도 있어 함께 경고.
- `추출/후보 < 0.7`(`PARSE_RATE_MIN`) → 행은 있는데 ID 추출 실패율이
  높음. `view('id')` 패턴 변경 등 부분 고장.
경고는 기존 "⚠️ 수집 실패" 이슈 경로로 보고. 첫 실행에서도 파서가
깨졌으면 baseline을 조용히 저장하지 않고 "첫 실행 — 파서 점검 필요"
이슈를 띄우도록 함.

**근거:** 스크래핑의 최대 약점은 학교가 마크업만 바꿔도 파서가 아무것도
못 잡으면서 **에러 없이 "신규 0건"처럼 보이는 조용한 실패**다. 기존
코드는 "0행 파싱"과 "새 글 0건"을 구분하지 못했다. 절대값 0만 보면
부분 고장(30행 → 3행)을 놓치므로, 한 실행 안에서 계산 가능한 **파싱
성공률**을 1순위 신호로 삼았다(게시판 크기와 무관해 신생 게시판 오탐도
적음). 게시판별 baseline(평균 행 수) 학습은 오탐이 실제 문제될 때 추가.

**남은 한계:** 마크업이 바뀌어도 '그럴듯한 잘못된 데이터'를 뽑는 경우
(제목 자리에 작성자명 등)는 성공률로 못 잡는다. 필드 형식 검증(날짜
꼴, 제목 비어있음)으로 일부 보완 가능하나 100%는 불가 — 정식 API가
없는 스크래핑의 근본 한계.

### 4. 실사이트 검증 → 썸네일 레이아웃 지원 + 건강검사 기준 교체

**계기:** 실제 ee.khu.ac.kr에 파서를 돌려보니(mock이 아니라 라이브)
`BMSR00040`(학사/취업/기타)은 30건 100% 추출됐지만 `BMSR00044`(소식/행사)는
**0건**이었다. 원인: 두 boardId의 목록 마크업이 다르다 — 전자는 표
(`tbody tr`), 후자는 **썸네일 카드**(`ul.bbs-thumb > li`, 제목 `strong.t`,
날짜 `span.date`). 3번 건강검사가 이 고장을 정확히 잡아낸 게 검증됨.

**바뀐 것 (A) 두 레이아웃 지원:** `fetch_board`가
`soup.select("tbody tr") or soup.select("ul.bbs-thumb > li")`로 표가 있으면
표를, 없으면 썸네일을 파싱. 이후 5개 게시판 전부 정상 추출 확인.

**바뀐 것 (B) 건강검사를 candidates → "전체 N 건" 기준으로 교체:**
3번의 candidates(데이터 행 수) 방식은 빈 게시판을 고장으로 오탐하는
문제가 있었다. 실제로 '행사'는 전체 0건인 빈 게시판인데 안내 li 하나가
잡혀 오탐이 났다. 그래서 `fetch_board`가 사이트의 `.bbs-total`("전체 N 건")을
함께 반환하고(`(rows, total)`), `main`은 기대치 `min(total, PAGE_SIZE)`의
`PARSE_RATE_MIN` 미만만 추출되면 경고하도록 변경. 전체 0건이면 검사를
생략해 빈 게시판 오탐을 없앴다. 두 레이아웃 모두 `.bbs-total`을 제공함을
확인함.

**검증:** 라이브 first-run에서 5개 게시판 baseline 저장 + 행사 오탐 없음,
재실행 시 신규 0건, 특정 게시판을 "전체 157건인데 0건 추출"로 시뮬레이션
시 "추출 수 급감" 경고 발생 — 모두 통과.

### 5. 테스트 인프라 — parse_board 분리 + fixture 테스트 + CI

**계기:** 테스트 부재가 가장 큰 구조적 공백이었다. `fetch_board`가
HTTP와 파싱을 한 함수에 묶어놔 CI 테스트가 원천 봉쇄돼 있었고, 4번의
"라이브 검증 완료"는 재현 불가능한 1회성 기록으로만 남아 있었다.

**바뀐 것:**
- `fetch_board`에서 파싱 본체를 **`parse_board(html) -> (rows, total)`
  순수 함수로 분리**(동작 무변경). `fetch_board`는 HTTP+재시도 후
  `parse_board(r.content)`를 호출하는 껍데기.
- `tests/fixtures/`에 라이브 검증된 마크업 구조를 본뜬 mock HTML 4종
  커밋: 표형(`table.html`) / 썸네일형(`thumb.html`) / 빈 게시판
  (`empty.html`, "게시물 없음" 행 포함) / 셀렉터 깨짐(`broken.html`,
  전체 152건인데 `onclick`이 `goView()`로 바뀜).
- `tests/test_parse_board.py` 4케이스: 표형·썸네일 추출(필드 정확성),
  빈 게시판은 건강검사 생략 조건(오탐 없음), 셀렉터 깨짐은
  `rate < PARSE_RATE_MIN` 판정에 걸림 — 판정식은 `main()`과 동일한
  식을 사용.
- `.github/workflows/ci.yml`: push(main)/PR에서 pytest 실행, 네트워크
  불필요. 기존 하드닝 원칙 그대로(액션 SHA 고정, `contents: read`,
  `persist-credentials: false`). pytest도 `requirements-dev.txt`로
  버전+sha256 해시 고정(pytest+하위 4개; dependabot pip 생태계가 갱신
  커버). `paths` 필터로 봇의 seen.json 커밋에는 CI가 돌지 않는다.

**근거:** 실사이트가 네트워크 차단 환경에선 접근 불가라 파서 검증이
매번 1회성 mock 스크립트로 반복돼 왔다. fixture를 레포에 커밋하면 그
검증이 재현 가능한 회귀 테스트가 되고, 이후 파서 수정(셀렉터 좁히기,
일반화 리팩터링)을 안전망 위에서 할 수 있다.

**검증:** 새 venv에서 `--require-hashes` 설치 성공(의존성 클로저 완전)
+ pytest 4/4 통과 + 분리 후 `fetch_board` 재시도·`is_error` mock 테스트
통과(동작 무변경 확인).

### 6. seen.json 키를 표시명 → 불변값(boardId:menuNo)으로

**바뀐 것:** seen.json의 게시판별 키(및 `__totals__` 내부 키)를 게시판
표시명("소식" 등)에서 `board_key(board, menu)` = `"BMSR00044:21700021"`
형태의 불변값으로 교체. `main()` 진입 시 `migrate_seen()`이 구버전 파일을
1회 이관한다 — 현재 `BOARDS` 표시명과 일치하는 옛 키를 새 키로 옮기고
(새 키가 이미 있으면 옛 키만 제거), 이관 후에는 no-op(멱등).

**근거:** 표시명 키는 이름만 바꿔도 옛 키가 찌꺼기로 남고 그 게시판의
알림 이력이 초기화됐다(item 3의 게시판 단위 baseline 덕에 "옛 글 폭발"은
이미 막혔지만, 이력 리셋 + 찌꺼기는 남는 문제). **일반화의 선행 조건** —
config에서 게시판 이름을 자유롭게 바꾸려면 상태 키가 이름과 무관해야 한다.
레포의 seen.json을 직접 고쳐 커밋하지 않고 런타임 이관으로 처리한 이유:
브랜치가 머지되기 전에도 봇이 main의 seen.json을 계속 갱신하므로, 파일을
미리 고치면 머지 충돌·이력 유실 위험이 있다. 이관 코드는 작고(10줄) 멱등이라
남겨둬도 부담 없음.

**검증:** `tests/test_seen_migration.py` 4케이스(이관·멱등·충돌 시 옛 키
제거·빈 파일) + mock 엔드투엔드(구버전 seen.json으로 `main()` 실행 →
키 이관 + 기존 글 800001 재알림 없음 + 신규 글만 알림) 통과.

### 7. Discord webhook 알림 채널 추가

**보안 전제(구현 규칙):**
- Webhook URL은 **Actions Secrets(`DISCORD_WEBHOOK`)로만 주입**한다.
  워크플로 파일·코드·커밋·채팅 어디에도 URL 평문을 두지 않는다
  (URL 소지 = 전송 권한이라 사실상 비밀번호다).
- 전송은 파이썬 안에서 `requests.post(url, json=...)`로 한다 — JSON
  직렬화가 이스케이프를 보장하고 셸을 거치지 않는다(이슈 파일 전달
  원칙의 자연스러운 확장).
- 실패 시 URL이 노출되지 않게 한다: requests 예외 메시지에는 URL이
  포함되므로 예외 본문을 찍지 않고 **예외 타입 + HTTP 상태 코드만** 출력.

**바뀐 것:** `notify_discord(title, body)` 추가, `write_output`에서
`has_new=True`일 때(신규 공지·수집 실패 이슈 모두) 호출. env가 비어
있으면(Secret 미설정) 조용히 생략 — Discord는 선택 기능. 페이로드는
embed 1개(`title` ≤256자, `description` ≤4096자로 절단) — 마크다운 링크
`[제목](URL)`은 plain content에선 렌더링되지 않고 embed에서만 되기 때문.
전송 실패는 삼키고 경고만 출력한다 — 주 채널(GitHub Issue)은 이 뒤의
워크플로 스텝이라, 여기서 예외가 새면 주 채널까지 죽는다.

**검증:** `tests/test_notify_discord.py` 3케이스(env 없으면 무전송 /
embed 페이로드·절단 / 실패 시 예외 미전파 + 출력에 URL·토큰 미노출).

## 미해결 / 다음 후보

- `is_hot` 부분일치 오탐(짧은 키워드가 단어 내부에 걸림).

- `PAGE_SIZE`(30) 초과분은 감시 못 함 — 3일에 30건 넘게 올라오는 게시판이
  있으면 놓칠 수 있음(현재 게시판 빈도상 문제 없음).
