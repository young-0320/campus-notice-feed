## 1. [보안] 워크플로 supply-chain 하드닝 3종 부재 (`khu-notice.yml`)

### 증상

이 봇은 3일마다 아무도 안 보는 새벽에 자동으로 도는 무인 job이고, 그 안에는
**레포에 push할 수 있는 열쇠**(GITHUB_TOKEN, `contents: write`)가 들어 있다.
이 열쇠를 도둑맞을 수 있는 문이 3개 열려 있었다.

**문 1 — 의존성을 "아무거나 최신"으로 설치.** `pip install requests
beautifulsoup4`는 실행할 때마다 PyPI에서 그 순간의 최신 버전을 받아온다.
requests가 내부적으로 쓰는 urllib3 같은 하위 패키지까지 합치면 8개인데,
이 중 **하나라도 악성 버전이 올라오면** 그 코드가 우리 job 안에서 그대로
실행된다. (PyPI 패키지 탈취 사건은 매년 실제로 일어난다.)

**문 2 — push 열쇠가 파일로 방치.** `actions/checkout`은 기본값
(`persist-credentials: true`)으로 push용 토큰을 `.git/config`에 **저장해
둔다**(마지막 스텝의 `git push`가 되는 이유). 그래서 문 1로 들어온 악성
코드가 그 파일 하나만 읽으면 열쇠를 얻는다. 열쇠를 얻으면
`check_notices.py`를 변조해 main에 push할 수 있고, 크론이 3일마다 변조된
코드를 다시 실행하므로 **감염이 영구 유지**된다. GITHUB_TOKEN은 워크플로
yml 수정만 막을 뿐(workflows 스코프 없음) 파이썬 스크립트 교체는 못 막는다.

**문 3 — 액션 버전이 "움직이는 이름표".** `@v4`/`@v5`/`@v7` 같은 git 태그는
주인이 다른 커밋으로 옮길 수 있다. 2025년 tj-actions 사건이 정확히 이
유형 — 인기 액션의 태그가 악성 커밋으로 옮겨져, 그 액션을 쓰던 수천 개
레포가 한 번에 감염됐다.

현실적 확률은 낮다(requests/bs4와 `actions/*`는 가장 감시가 심한
패키지들). 즉 긴급 패치가 아니라 "비용이 거의 0이라 안 할 이유가 없는
보험"에 해당한다.

### 수정

기능 변화는 0이고, 자물쇠 3개를 추가했다.

1. **`requirements.txt` 신설 + `pip install --require-hashes`.** 8개
   패키지(직접 2 + 하위 6)를 버전 고정하고, PyPI가 공개하는 릴리스 파일별
   sha256 해시를 전부 수록했다. 지문이 다르면 설치 자체가 거부되므로
   오염된 신규 릴리스가 유입될 수 없다.
2. **checkout에 `persist-credentials: false` 명시.** pip 코드가 실행되는
   동안 디스크에 토큰이 없다. 토큰이 꼭 필요한 마지막 push 스텝에서만
   `env`로 주입해 `git push "https://x-access-token:…"` 형태로 쓴다.
   (완벽한 벽은 아니다 — 이론상 악성 코드가 push 스텝에서 실행될 git
   hook을 미리 심을 수는 있다. 다만 "파일 하나 읽으면 끝"을 "표적형
   다단계 공격 필요"로 난이도를 올리는 표준 하드닝.)
3. **세 액션을 40자 커밋 SHA로 고정**(`# v4` 주석 병기) + `.github/
   dependabot.yml` 추가. 태그가 어디로 옮겨지든 검증된 커밋만 쓰고,
   버전을 못박아서 놓치게 되는 업데이트는 dependabot이 월 1회 PR로
   제안한다(pip 해시 갱신 포함).

### 검증

- 새 venv에서 `pip install --require-hashes -r requirements.txt` 실제 설치
  성공 — 의존성이 하나라도 누락되면 이 옵션이 즉시 실패하므로, 8개로
  의존성 클로저가 완전함이 확인됨.
- 고정된 bs4 4.13→4.15 상향이 포함되므로, 표형·썸네일형 mock HTML로
  `fetch_board` 파싱 확인 — 두 레이아웃 모두 id/제목/날짜/총계 정상 추출.
- 워크플로·dependabot YAML 파싱 통과. 액션 SHA는 GitHub API로 태그를
  직접 조회해 확보(v4=`34e11487…`, v5=`a26af69b…`, v7=`f28e40c7…`).

## 2. [버그] seen.json 200개 트림이 문자열 역순 정렬 (`check_notices.py:194`)

### 증상

`seen.json`은 게시판별로 이미 알림 보낸 글 ID를 저장하고, 파일이 무한히
커지지 않도록 매 실행마다 "최신 200개"만 남기고 잘라낸다. 그런데 이
트림이 다음 코드처럼 **문자열 정렬**로 되어 있었다.

```python
seen[name] = sorted(known | {r["id"] for r in rows}, reverse=True)[:200]
```

게시글 ID는 숫자지만 문자열로 다루기 때문에, 자릿수가 달라지는 순간
정렬이 숫자 크기와 어긋난다. 예를 들어 6자리 `"999999"`와 7자리
`"1000000"`을 문자열로 비교하면 첫 글자 `'9' > '1'`이라 `"999999"`가
더 크다고 판정된다.

재현:

```python
sorted({"999998", "999999", "1000000", "1000001"}, reverse=True)[:2]
# → ['999999', '999998']  ← 7자리 신규 ID가 탈락
```

즉, ID가 6→7자리로 넘어간 뒤 누적 ID가 200개를 넘으면, **가장 최신인
7자리 ID들이 오히려 먼저 잘려나간다.** 신규 글이 `seen`에 저장되자마자
다음 트림에서 축출되므로, 매 실행마다 같은 글이 "신규"로 재판별되어
**동일 공지가 영구적으로 반복 알림**되는 상태에 빠진다. (현재 ID는
5자리라 아직 발현 전이지만, 5→6자리 전환에서도 동일하게 터진다.)

### 수정

정렬 키를 정수로 바꿔 숫자 크기 기준으로 최신 200개를 유지하도록 했다.

```python
seen[name] = sorted(known | {r["id"] for r in rows}, key=int, reverse=True)[:200]
```

`key=int`가 안전한 근거: ID는 `ID_RE`(`view\(\s*['"](\d{4,})['"]`)로만
추출되므로 항상 숫자 문자열이고, 기존 `seen.json`에 저장된 값도 전부
숫자임을 확인했다(비숫자 항목 0개). 수정 후 위 재현 케이스는
`['1000001', '1000000']`으로 최신 ID가 올바르게 유지되며,
`py_compile`도 통과했다.

## 3. [버그] 전역 `first_run` 판정 — 게시판 추가·부분 실패 시 옛 글 폭발 (`check_notices.py:161,187`)

### 증상

이 봇은 "이전에 본 적 없는 글 = 신규"로 판별해 알림을 보낸다. 처음
돌릴 때는 게시판의 기존 글이 전부 "신규"가 되어 알림이 폭주하므로,
**첫 실행에서는 기존 글을 `seen.json`에 baseline으로만 저장하고 알림은
보내지 않는** 장치가 있다. 그런데 이 "첫 실행" 판정이 **레포 전체 기준
하나의 전역 플래그**로만 되어 있었다.

```python
first_run = not seen          # seen.json 전체가 비었을 때만 True
...
if not first_run:             # 이때만 신규 알림 수집
    for r in fresh:
        ...
```

`seen`은 게시판별 딕셔너리인데, `first_run`은 그 **전체**가 비었는지만
본다. 그래서 "이 개별 게시판을 이전에 본 적이 있는가"는 전혀 따지지
않는다. 이 때문에 이미 baseline이 있는 상태(`first_run == False`)에서
어떤 게시판의 `known`이 비어 있으면, 그 게시판의 목록 전체(최대
`PAGE_SIZE`=30건)가 한꺼번에 "신규"로 알림된다. 실제로 이 조건에
빠지는 경로가 둘 있다.

1. **게시판 추가.** README가 안내하는 대로 `BOARDS`에 게시판을 하나
   추가하면, 이미 `seen.json`이 존재하므로 `first_run`은 `False`다.
   하지만 새 게시판 이름은 `seen`에 없어 `known`이 비고, 그 게시판의
   기존 글 30건이 전부 신규 알림으로 터진다.
2. **첫 실행 중 일부 게시판 fetch 실패.** 진짜 첫 실행에서 한 게시판만
   네트워크 예외로 빠지면(`continue`), 그 게시판 키 없이 `seen.json`이
   저장된다. 다음 실행은 `first_run == False`가 되고, 실패했던 게시판은
   이제 성공하지만 `known`이 비어 역시 옛 글이 폭발한다.

이는 `log.md`에 이미 1순위로 적힌 "게시판 표시명을 바꾸면 그 게시판이
통째로 신규로 폭발" 버그와 **정확히 같은 기제**다. 표시명 변경만
트리거로 기록돼 있었을 뿐, 게시판 추가·부분 실패라는 다른 두 트리거는
누락돼 있었다.

### 수정

알림 게이트를 전역 플래그에서 **게시판 단위 판정**으로 바꿨다.

```python
# 이 게시판을 이전 실행에서 본 적이 있을 때만 신규 알림.
# name not in seen = 첫 실행이거나 새로 추가된 게시판 → baseline만 저장.
if name in seen:
    for r in fresh:
        ...
```

`if not first_run` → `if name in seen`으로의 교체다. 제안됐던
`if not first_run and name in seen` 대신 `name in seen` 하나만 쓴
이유: `first_run`이 참이면 `seen`이 비어 있어 모든 게시판에서
`name in seen`이 어차피 거짓이 된다. 즉 `not first_run`은 완전히
중복이고, `name in seen`은 그 자체로 "이 게시판의 baseline이 이미
있는가"라는 정확한 의도를 표현한다. (참고로 게이트 검사 시점(187행)은
`seen[name]`이 저장되는 194행보다 앞이므로, 현재 게시판의 `name in
seen`은 항상 **이전 실행까지의** 상태를 반영한다.) `first_run` 변수는
첫 실행 안내 메시지 출력 분기(198행)에서 계속 쓰이므로 그대로 뒀다.

### 검증

`fetch`/네트워크를 제거하고 알림 게이트 로직만 격리해 5개 시나리오를
재현, 전부 기대대로 동작함을 확인했다(`py_compile`도 통과).

- A. 진짜 첫 실행(seen 빔) → 알림 0, baseline 저장.
- B. 재실행 중 한 게시판에 신규 1건 → 그 1건만 알림.
- C. 운영 중 게시판 추가 → **알림 0**(옛 글 폭발 없음), 새 게시판
  baseline만 저장.
- D. 추가됐던 게시판에 이후 신규 1건 → 그 1건만 알림(정상 전환).
- E. 첫 실행 중 한 게시판 fetch 실패 → 그 게시판이 성공하는 다음
  실행에서도 **알림 0**(폭발 없음), baseline만 저장.

## 4. [버그] 이슈 생성 → push 순서인데 pull 없는 push — 실패 시 중복 이슈 (`khu-notice.yml:61`)

### 증상

워크플로는 ① 신규 공지 이슈를 먼저 발행하고 ② 그 다음에 `seen.json`을
커밋·push한다. 그런데 push가 **pull/rebase 없이** 단발로만 실행됐다.

```yaml
git push "https://...@github.com/${{ github.repository }}.git" "HEAD:${{ github.ref_name }}"
```

checkout 시점과 push 시점 사이에 누군가 main에 커밋을 올리면 — README가
`keywords.txt`를 GitHub 웹에서 직접 편집하라고 안내하고 있어 충분히
현실적인 시나리오다 — 봇의 push는 non-fast-forward로 거부된다. 문제는
순서다. **이슈는 이미 발행된 뒤라 되돌릴 수 없는데, "이 글들을 알렸다"는
기록(`seen.json`)만 유실된다.** 다음 실행은 같은 글들을 다시 신규로
판별해 **동일 공지 이슈가 중복 발행**된다.

기존의 `concurrency` 그룹(9–12행)은 스케줄 실행과 수동 실행이 겹치는
것만 막는다. 워크플로 밖에서 일어나는 사람의 직접 커밋과의 경합은 막지
못한다.

### 수정

push 전에 `git pull --rebase`로 원격의 새 커밋을 흡수하고, 실패 시
최대 3회 재시도하도록 했다.

```yaml
REMOTE="https://x-access-token:${GITHUB_TOKEN}@github.com/${{ github.repository }}.git"
for i in 1 2 3; do
  if git pull --rebase "$REMOTE" "${{ github.ref_name }}" \
     && git push "$REMOTE" "HEAD:${{ github.ref_name }}"; then
    exit 0
  fi
  echo "push 실패 (시도 $i/3)"
  sleep 5
done
exit 1
```

설계 근거:

- **rebase 충돌 걱정 없음.** `seen.json`은 봇만 커밋하는 파일이라 사람
  커밋(keywords.txt·README 등)과 같은 파일을 건드릴 일이 사실상 없다.
- **pull에도 토큰 URL 사용.** checkout이 `persist-credentials: false`라
  origin에 자격증명이 없으므로, pull도 push와 같은 토큰 포함 URL로 한다
  (토큰을 `.git/config`에 남기지 않는 기존 보안 원칙 유지).
- **shallow clone(depth 1)에서도 rebase 동작.** 분기점은 항상 checkout
  시점의 커밋(로컬에 존재)이고 그 이후 커밋만 fetch되므로 merge-base가
  보장된다.
- **3회 모두 실패하면 `exit 1`**로 스텝을 실패시켜 문제가 조용히 묻히지
  않게 했다.

### 검증

- 워크플로 YAML 파싱 통과.
- 재시도 루프를 `bash -e`(Actions run 스텝과 동일한 셸 옵션)에서
  시뮬레이션: 즉시 성공 → exit 0, 2번째 시도 성공 → exit 0, 3회 전부
  실패 → "시도 1/3~3/3" 출력 후 exit 1. `set -e` 환경에서 실패한
  push가 스크립트를 중단시키지 않고 재시도로 이어지는 것 확인.
  (남는 잔여 리스크: pull 시점~push 시점 사이의 초단위 경합은 재시도가
  흡수하고, 이슈 발행 자체와 push 사이의 근본적 비원자성은 순서를
  바꾸지 않는 한 남지만 재시도로 실패 확률을 크게 낮췄다.)

## 5. [버그] '수집 실패' 이슈 dedup 부재 — 3일마다 무한 생성 (`check_notices.py:208`, `khu-notice.yml:45`)

### 증상

문제가 두 겹이었다.

**① 같은 실패 이슈가 계속 쌓인다.** 사이트 마크업이 영구적으로 바뀌면
(셀렉터가 더 이상 안 맞으면) 매 실행이 "수집 실패"를 보고하는데,
워크플로는 무조건 `issues.create`만 호출한다. 실패 이슈의 제목은
`[KHU EE] 공지 수집 실패`로 고정이므로, 사용자가 파서를 고칠 때까지
**내용이 똑같은 이슈가 3일마다 하나씩 무한 생성**된다. 신규 공지
이슈는 내용이 매번 다르니 여러 개가 정당하지만, 실패 이슈는 "지금
고장 나 있다"는 상태 알림이라 open 1개면 충분하다.

**② 일회성 네트워크 흔들림에도 실패 이슈가 나간다.** `requests.get`이
재시도 없이 한 번만 시도하므로, 대학 서버가 한 번 타임아웃하거나
5xx를 돌려주기만 해도 곧바로 수집 실패 이슈가 발행된다. 진짜 고장
(마크업 변경)과 일시 장애가 구분되지 않아 실패 이슈의 신호 가치가
떨어진다.

### 수정

**① 실패 이슈 dedup — 동일 제목의 open 이슈가 있으면 댓글로.**

파이썬 `write_output`에 `is_error` 플래그를 추가해, 실패 이슈("수집
실패"·"첫 실행 파서 점검")일 때 `GITHUB_OUTPUT`으로 `is_error=true`를
내보낸다. 워크플로 github-script는 이 플래그가 참일 때만
`issues.listForRepo({labels: 'notice', state: 'open'})`로 open 이슈를
조회해, **제목이 정확히 같은** 이슈가 있으면 `issues.create` 대신
`issues.createComment`로 그 이슈에 댓글을 단다(재발 사실은 계속
기록됨). 사용자가 파서를 고치고 이슈를 close하면 다음 실패는 다시 새
이슈로 열린다.

설계 선택 두 가지:

- **dedup은 실패 이슈에만 적용.** 제목 일치만으로 모든 이슈를 dedup하면
  정상 공지 이슈도 걸릴 수 있다 — 제목이 `신규 공지 N건` 형식이라,
  이전 공지 이슈가 open인 채로 건수가 우연히 같으면 **다른 내용의 새
  공지가 옛 이슈의 댓글로 숨는** 오동작이 된다. `is_error`로 범위를
  한정해 이를 차단했다.
- **인젝션 방지 원칙 유지.** 기존 설계는 "제목/본문은 파일로, `${{ }}`
  에는 불리언만"인데, `is_error`는 파이썬이 만드는 `true`/`false`
  불리언이라 `${{ }}`로 넘겨도 안전하다(`has_new`와 동급). 제목 비교는
  JS 안에서 파일로 읽은 값으로 하므로 게시글 텍스트는 여전히 `${{ }}`
  치환을 거치지 않는다.

**② `fetch_board`에 재시도 추가.**

`requests.get` + `raise_for_status`를 최대 3회 시도(실패 시 3초 간격,
즉 2회 재시도)로 감쌌다. `requests.RequestException`(타임아웃·연결
오류·HTTP 오류 전부 포함)만 재시도하고, 3회째 실패면 예외를 그대로
전파해 기존 에러 경로(게시판별 `수집 실패` 항목)로 합류한다.

### 검증

전부 네트워크 없이 mock으로 재현했다.

- **재시도**: `requests.get`을 "2번 타임아웃 후 성공"으로 mock →
  예외 없이 정상 반환(호출 정확히 3회). "항상 실패"로 mock → 3회
  시도 후 `Timeout` 전파(무한루프 없음).
- **is_error 출력**: 에러/정상 공지/신규 없음 3케이스 모두
  `GITHUB_OUTPUT`에 `has_new`·`is_error`가 기대값으로 기록됨.
- **워크플로 JS**: script 블록 로직을 node로 시뮬레이션 — (1) 동일
  제목 open 실패 이슈 존재 → 그 이슈에 댓글, (2) 실패 이슈 최초 발생
  → 새 이슈 생성, (3) 정상 공지는 제목이 우연히 겹쳐도 항상 새 이슈.
  세 경로 모두 기대대로 동작.
- `py_compile`·워크플로 YAML 파싱 통과.

## 6. [취약가정] 파서 건강검사 0.7 임계값의 사각지대 (`check_notices.py` 건강검사부)

### 배경

건강검사는 "사이트가 표시하는 전체 건수(`.bbs-total`의 '전체 N 건')"와
"실제로 파싱해낸 글 수"를 비교해, 마크업이 바뀌어 파서가 조용히 실패하는
것(에러 없이 '신규 0건'처럼 보임)을 잡아내는 장치다. 기존 로직은 이랬다.

```python
expected = min(total, PAGE_SIZE) if total is not None else None
if expected:  # 0(빈 게시판)·None(총계 미확인)이면 비율 검사 생략
    if len(rows) < expected * PARSE_RATE_MIN:      # PARSE_RATE_MIN = 0.7
        errors.append("추출 수 급감 ...")
elif total is None and not rows:
    errors.append("총계·글 모두 파싱 실패 ...")
```

즉 **전체 건수(total)를 읽을 수 있을 때만** 비율 검사가 작동하고, total을
못 읽으면(`None`) `rows`까지 0일 때만 경고했다. 여기에 세 가지 사각지대가
있었다.

- **① 부분 고장(추출률 70~99%).** 30건 중 24건만 파싱돼도 `24 ≥ 30×0.7`
  이라 임계값 위 → 무경고. 매 실행 같은 6건을 결정적으로 놓치면 그 글들은
  영원히 알림되지 않는 **조용한 알림 유실**이 된다.
- **② `.bbs-total` 셀렉터만 깨짐.** 사이트가 전체 건수 표시 부분의 마크업만
  바꾸면 `total=None`이 된다. 이때 `rows`는 여전히 잡히므로(리스트 마크업은
  멀쩡) 위 코드의 `if expected:`도 `elif ... and not rows`도 **둘 다 건너뛰어
  비율 검사가 통째로 사라진다.** 이 상태에서 리스트 파싱까지 부분 고장이
  겹치면(②+①) 아무 경고 없이 글이 유실된다.
- **③ 위 ②의 부분집합** — `.bbs-total` 소실 + 리스트 부분 파싱 조합이
  완전 무경고라는 것.

핵심은 **비교 기준(total)이 사라지면 검사 자체가 무력화**된다는 점이었다.

### 수정

게시판별로 **지난 성공 실행의 전체 건수를 `seen.json`에 기억**해 두고,
이번 실행에서 total을 못 읽으면 그 이전 값과 비교해 "이전엔 읽혔는데 지금
안 읽힘 = `.bbs-total` 셀렉터가 깨진 것"으로 판정한다.

`seen.json`에 게시판명과 겹치지 않는 메타 키(`__totals__`)를 두어
`{게시판명: 마지막_전체건수}`를 저장한다. 판정 로직은 이렇게 바뀌었다.

```python
prev_total = totals.get(name)
if total is not None:
    totals[name] = total                    # 성공 → 기준값 갱신
    expected = min(total, PAGE_SIZE)
    if expected and len(rows) < expected * PARSE_RATE_MIN:
        errors.append("추출 수 급감 ...")     # 기존 비율 검사(사각지대 ①은 한계로 잔존)
elif prev_total:
    errors.append("전체 건수(.bbs-total) 파싱 실패 (지난 N건 → 이번 미확인)")  # ② ③ 신규 감지
elif not rows:
    errors.append("총계·글 모두 파싱 실패 ...")  # 이력 없는 첫 목격은 절대값으로만 판단
```

설계 근거:

- **`__totals__`는 별도 메타 키.** 게시판 ID 목록(`seen[name]`은 리스트)
  구조와 200개 트림 로직을 건드리지 않으려고, 총계는 `seen["__totals__"]`
  딕셔너리에 분리 저장했다. 게시판 표시명과 절대 겹치지 않게 밑줄로 감쌌고,
  알림 게이트(`if name in seen`)와 `first_run` 판정에도 영향을 주지 않는다
  (`__totals__`는 성공한 total이 하나라도 있을 때만 기록되므로, 이 키가
  존재하면 반드시 대응하는 게시판 키도 존재 → `first_run = not seen`이
  여전히 정확).
- **`prev_total`은 truthy로 검사(0 제외).** 빈 게시판(전체 0건)은 애초에
  놓칠 글이 없어 사각지대 ②의 대상이 아니다. 전체 0건이던 게시판의
  `.bbs-total`이 나중에 깨져도 별도 경고 없이 기존 "총계·글 모두 실패"
  경로(rows까지 0일 때)로만 잡히게 해서, 빈 게시판 오탐을 막았다.
- **사각지대 ①(부분 고장 70~99%)은 근본 해결 불가로 남긴다.** 정식 API
  없이 스크래핑만으로는 "그럴듯한 잘못된 데이터"를 구분할 수 없다는 점은
  `log.md`에 이미 한계로 기록돼 있다. 이번 수정은 ②·③(비교 기준 소실로
  인한 검사 무력화)만 메운다.

### 검증

전부 네트워크 없이 재현했다.

- **판정 로직 단위 시뮬 10케이스** — 정상 추출/빈 게시판/소량 게시판은
  무경고, 부분 고장 60%는 경고·80%는 무경고(한계 확인), `total=None +
  이전값 있음`은 rows 유무와 무관하게 경고(②③), `total=None + 이전값 없음`은
  rows 0일 때만 경고(첫 목격 절대값 판정), 빈 게시판(prev 0)에서 total 소실은
  오탐 없이 기존 경로로만 처리 — 전부 기대대로.
- **모듈 통합 재현(30행 기준)** — Run1 정상 첫 실행에서 `__totals__` 5개
  게시판 저장 + 알림 없음, Run2에서 `소식`의 total만 `None`으로 만들었을 때
  **`소식` 한 건만** "전체 건수(.bbs-total) 파싱 실패 (지난 실행 30건 →
  이번엔 미확인)" 경고(나머지 4개 무경고, 오탐 없음), Run3 정상 복구 후
  신규 1건만 정상 알림. `__totals__`가 알림 게이트·`first_run`을 오염시키지
  않고 게시판 키는 계속 리스트로 유지됨을 확인.
- `py_compile` 통과.

## 9. [보안/low] 제목·예외 메시지의 마크다운 미이스케이프 (`check_notices.py:246,253,185`)

### 증상

이슈 본문은 게시글 제목을 마크다운 링크로 조립한다.

```python
lines.append(f"- **[{r['board']}]** [{r['title']}]({r['url']}) `{r['date']}`")
```

제목은 학과 사이트에서 긁어온 **신뢰 불가 텍스트**인데 이스케이프 없이
`[링크텍스트](URL)` 구문 안에 들어간다. 만약 게시글 제목 자체에 마크다운
특수문자가 들어 있으면 — 예를 들어 제목이 `등록금 안내](https://evil.example)
[진짜공지` 라면 — 렌더된 이슈에서 `]`가 링크를 조기 종료시켜 **"등록금
안내"라는 링크의 목적지가 피싱 사이트로 위조**된다. 이슈가 메일 알림으로도
가므로 메일에서도 동일하게 위조된다. 예외 메시지(`{e}`, 185행)도 서버
응답 내용이 섞일 수 있는 신뢰 불가 텍스트인데 무가공으로 삽입됐다.

심각도는 **low**가 맞다(검토 과정에서 high 주장 → 스켑틱 2명이 과장으로
판정). 근거: ① 악용하려면 학과 게시판에 글을 올릴 권한이나 사이트 침해가
필요한데, 그 시점이면 공격자는 이미 진짜 공지 채널을 쥔 상태다. ② 수신자가
사실상 레포 소유자 1인이다. ③ 코드 실행이 아니라 링크 위조에 그친다.
④ URL 자체는 `ID_RE`가 숫자만(`\d{4,}`) 추출해 조립하므로 안전하다.
다만 수정 비용이 거의 0이고, 향후 타 학과/학교로 일반화하면서 수신자가
늘거나 URL에 인증 파라미터가 붙으면 실질 위험으로 승격되므로 지금 막았다.

### 수정

마크다운 링크/코드 구문에 쓰이는 4개 문자를 역슬래시로 이스케이프하는
`md_escape`를 추가하고, 신뢰 불가 텍스트가 본문에 들어가는 네 지점(제목
×2, 날짜 ×2 자리)과 예외 메시지에 적용했다.

```python
def md_escape(text) -> str:
    return re.sub(r"([\\\[\]`])", r"\\\1", str(text))
```

- 적용 지점: hot/rest 목록의 `title`·`date`, 수집 실패 메시지의 `{e}`.
- `board`(BOARDS 상수)·URL(숫자 ID로만 조립)·건수 등 신뢰 가능한 값은
  건드리지 않았다.
- 이스케이프 문자를 `` \ [ ] ` `` 4개로 최소화한 이유: 링크 위조에 필요한
  건 `[`/`]`뿐이고, `` ` ``는 코드 스팬 탈출, `\`는 이스케이프 자체의
  무력화를 막기 위해 필요하다. `*`·`_` 같은 서식 문자까지 이스케이프하면
  정상 제목이 `\*` 투성이로 지저분해지는데, 서식 깨짐은 보안 문제가
  아니므로 제외했다.

### 검증

- 위 위조 예시 제목으로 렌더 문자열 생성: 수정 전에는 `[등록금
  안내](https://evil.example/phish)`라는 피싱 링크가 만들어졌으나, 수정
  후에는 `\]`·`\[`로 무력화되어 **제목 전체가 진짜 공지 URL을 가리키는
  하나의 링크 텍스트**로만 렌더된다.
- ``md_escape("a]b[c`d\e")`` → ``a\]b\[c\`d\\e`` 등 단위 케이스 통과,
  특수문자 없는 정상 제목은 변형 없음 확인. `py_compile` 통과.

## 10. [버그/low] 파서 고장 시 결함 baseline 저장 — log.md 서술과 코드 불일치 (`check_notices.py` 건강검사·baseline 갱신부)

### 증상

`log.md`(43–45행)는 이렇게 약속한다: "첫 실행에서도 파서가 깨졌으면
baseline을 조용히 저장하지 않고 '첫 실행 — 파서 점검 필요' 이슈를
띄운다." 그런데 코드에서 `save_seen`은 `first_run` 분기보다 **먼저,
무조건** 실행됐다. 즉 이슈는 뜨지만 **결함 있는 추출 결과도 그대로
baseline으로 커밋**되고 있었다 — 문서가 코드보다 앞선 상태.

왜 문제인가: 파서가 반쯤 고장 난 상태(예: 30건 중 3건만 추출)로 첫
실행이 돌면, 그 3건만 "이미 본 글"로 저장된다. 나중에 파서를 고치는
순간 나머지 27건의 **옛 글이 전부 "신규"로 오알림**된다. 완전 고장(0건
추출)이면 빈 목록이 baseline으로 굳어 게시판 전체(최대 30건)가 터진다.
항목 3(옛 글 폭발)과 같은 뿌리의 버그다.

심각도는 low가 맞다(스켑틱이 high → 과장 판정). 현 배포는 이미 정상
baseline이 커밋돼 있어 first_run이 재발하는 건 seen.json 리셋이나 타
학과/학교 일반화 시점뿐이고, 결과도 1회성 알림 노이즈이지 데이터 유실이
아니다. 다만 일반화가 이 프로젝트의 명시된 방향이라 그 시점에 밟을
지뢰를 미리 제거했다.

### 수정

"코드와 log.md 중 하나를 맞춰라"에서 **코드를 문서(의도)에 맞췄다.**
전체 저장을 통째로 스킵하는 대신, 건강검사에 걸린 게시판만 **게시판
단위로** baseline 생성에서 제외했다 — 정상인 게시판의 baseline까지
버릴 이유가 없기 때문이다.

```python
parse_suspect = False   # 건강검사 3개 분기(급감/총계 실패/전무)에서 True로
...
# 건강검사에 걸린 게시판의 baseline은 새로 만들지 않는다.
# 이미 baseline이 있는 게시판은 known과의 합집합이라 잃을 게 없으므로 갱신.
if name in seen or not parse_suspect:
    seen[name] = sorted(known | {r["id"] for r in rows}, key=int, reverse=True)[:200]
```

동작 정리:

- **첫 실행(또는 신규 게시판) + 파서 의심** → 그 게시판 키를 저장하지
  않음. 다음 실행에서 파서가 정상이면 항목 3의 `name in seen` 게이트
  덕에 **알림 폭발 없이** baseline만 조용히 저장된다(자연 수렴).
- **기존 게시판 + 파서 의심** → 기존 동작 유지. `known`과의 합집합이라
  결함 추출본이 섞여도 기존 기록을 잃지 않는다.
- fetch 예외 경로는 원래 `continue`로 저장을 건너뛰므로 변화 없음.

### 검증

  필요" 이슈, `seen.json`에 A만 저장되고 **B 키 없음**.
- F2. 다음 실행에서 B 복구 → **알림 0**(옛 글 폭발 없음), B baseline 저장.
- F3. 운영 중 A 추출 수 급감(30건 중 1건) → 급감 경고 발행, 기존
  baseline(합집합) 보존.
- F4. 회귀: 정상 신규 1건은 여전히 알림됨.

## 11. [취약가정/low] `r.text` 인코딩 폴백 — charset 헤더 사라지면 한글 전멸 (`check_notices.py:119`)

### 증상

파싱이 `BeautifulSoup(r.text, ...)`로 되어 있었다. `r.text`의 인코딩
판정은 requests가 하는데, HTTP 표준의 낡은 규칙 때문에 **서버가
`Content-Type` 헤더에서 `charset`을 빼는 순간 ISO-8859-1(서유럽
문자셋)로 폴백**한다. UTF-8 한글을 ISO-8859-1로 읽으면 제목이 전부
`ë°ëì²´ ì¥íê¸...` 같은 mojibake가 된다.

더 나쁜 건 **건강검사가 이를 통과시킨다**는 점이다. 검사는 추출 "개수"만
보는데 mojibake여도 ID·행 개수는 멀쩡히 잡히므로, 에러 하나 없이 깨진
제목으로 이슈가 발행된다(⭐ 키워드 매칭도 전부 무력화). 지금은 서버가
charset을 보내고 있어 잠재 상태지만, 학교 서버 설정 변경 한 번이면
발현하는 취약 가정이다.

### 수정

한 줄 — 문자열(`r.text`) 대신 바이트(`r.content`)를 넘긴다.

```python
soup = BeautifulSoup(r.content, "html.parser")
```

bs4는 바이트를 받으면 문서 안의 `<meta charset>` 선언으로 인코딩을
직접 판정하므로, HTTP 헤더에 charset이 있든 없든 결과가 같다. 인코딩의
진실은 문서 자신이 제일 잘 알고 있으니 판정 주체를 requests(헤더
의존)에서 bs4(문서 의존)로 옮긴 것이다.

### 검증

requests의 실제 폴백 경로(`get_encoding_from_headers`)로 시나리오 재현:
UTF-8 한글 문서 + `Content-Type: text/html`(charset 없음) 응답을 만들어
확인.

- 수정 전 경로(`r.text`): 제목이 `ë°ëì²´ ì¥íê¸ ìë´`로 깨짐 —
  버그 재현.
- 수정 후(`r.content`): `fetch_board`가 "반도체 장학금 안내"·total 1을
  정상 반환. 기존 표형·썸네일형 mock 파싱도 회귀 없음.
